# AGENTS.md

## 1. 目的

本文件用于约束 AI 在本项目中的行为，目标是：

- 节约 token / 额度消耗
- 避免过度实现
- 避免大范围重构
- 避免无关工具调用
- 避免启动项目、截图、全量测试等高消耗操作
- 避免覆盖用户已有改动

AI 每次开始任务前必须先读取并遵守本文件。

---

## 2. 项目基本信息

本项目是前后端分离项目：

- `backed`：Spring Boot 后端
- `fronted`：Vue 3 前端
- 项目类型：个人决策回测器 / Decision Echo

如果需要了解项目业务背景，可读取 `PROJECT_HANDOFF.md` 中与当前任务相关的部分。

不要在每次任务中重复总结整个项目背景。

---

## 3. 总体原则

默认遵守以下原则：

1. 最小改动。
2. 只修改与当前任务直接相关的文件。
3. 不要自行扩展未要求的功能。
4. 不要大范围重构。
5. 不要重写整个页面或模块。
6. 不要引入大型依赖。
7. 不要修改无关文件。
8. 不要覆盖用户已有改动。
9. 不要输出长篇计划，除非用户明确要求。
10. 不要输出完整 diff，除非用户明确要求。
11. 不要把工具日志大段粘贴到回复中。
12. 完成后只简洁说明：改了什么、验证了什么、还有什么需要用户处理。

---

## 4. 任务边界

开始任务前，先判断任务类型：

- 只做计划：只输出计划，不改代码。
- 后端任务：只修改 `backed`。
- 前端任务：只修改 `fronted`。
- UI 任务：只修改相关 Vue 文件和 CSS，不改接口和业务逻辑。
- 联调任务：只有用户明确要求时，才同时改前端和后端。

如果用户没有明确要求同时改前后端，默认不要跨端修改。

如果发现任务必须跨端完成，先说明原因，并建议拆成：

1. 后端接口
2. 前端接入
3. 最小验证

---

## 5. 修改前检查

修改代码前只做必要检查：

```bash
git status
```

然后只读取当前任务相关文件。

不要为了“全面了解项目”而扫描大量无关文件。

优先读取：

- 当前页面或模块文件
- 当前接口文件
- 当前 service / controller / mapper
- 与报错直接相关的文件
- 用户指定的文件

---

## 6. 后端规则

后端目录：`backed`

后端开发时：

1. 保持 Spring Boot 现有结构。
2. 遵守 controller / service / mapper 分层。
3. Controller 只做参数接收和结果返回。
4. Service 负责业务校验、事务和业务规则。
5. Mapper 使用 MyBatis 注解 SQL。
6. 所有用户数据必须按 `userId` 隔离。
7. 不随意修改已有登录逻辑。
8. 不随意修改数据库结构。
9. 不新增无关接口。
10. 不把 mock 数据替代真实接口。

新增或修改接口时，先明确接口契约：

- 请求方法
- 请求路径
- 请求字段
- 响应字段
- 是否需要 token
- 主要错误处理

---

## 7. 前端规则

前端目录：`fronted`

前端开发时：

1. 保持 Vue 3 + Vite + Pinia + Element Plus 现有结构。
2. 优先修改相关页面、API 文件、store 文件和样式文件。
3. 不随意重构路由。
4. 不随意修改登录态逻辑。
5. 不新增后端接口假设。
6. 接口必须来自已有代码、用户提供的接口契约，只有必要时创建新的接口。
7. UI 调整不要影响业务逻辑。
8. 不要为了美化重写整个页面。
9. 不要新增复杂动画库。
10. 移动端不能出现横向滚动。

---

## 8. UI 风格规则

项目 UI 风格倾向：

- 年轻化
- 扁平化
- 浅色
- 偏个人应用
- 偏大众用户
- 轻动态
- 柔和圆角和阴影

避免：

- 企业后台管理风
- 商务 SaaS 风
- 深色厚重风
- 过度营销 landing page
- 过度技术感文案
- 无意义 hero 区
- 大面积强渐变

Dashboard 优先级：

1. 过去记录
2. 待回看内容
3. 回测结果
4. 统计信息
5. 新建 / 编辑 / 操作入口

新建、编辑、补回测、修改密码等操作应使用 Modal / Dialog / 表单，不要占据主内容。

---

## 9. 验证规则：默认轻量验证

为了节约 token，默认只做最小验证。

### 默认不要做

除非用户明确要求，否则不要：

- 启动前端服务
- 启动后端服务
- 打开浏览器
- 截图
- Playwright 检查
- 移动端截图检查
- 完整人工验证码登录流程
- 新增复杂 E2E 测试
- Git 提交
- Git 推送
- 跑全量测试
- 扫描大量无关文件
- 输出大量日志

### 前端默认验证

前端任务默认只运行：

```bash
cd fronted
npm run build
```

### 后端默认验证

后端任务优先只运行相关测试类，例如：

```bash
cd backed
.\mvnw.cmd -Dtest=AuthServiceImplTest test
```

只有在以下情况才运行后端全量测试：

- 用户明确要求
- 改动影响多个后端模块
- 局部测试无法覆盖明显风险

后端全量测试命令：

```bash
cd backed
.\mvnw.cmd test
```

### 浏览器验证规则

浏览器验证只有在用户明确要求时才做。

即使修改了前端布局、弹窗、交互或响应式，默认也只运行 `npm run build`。

如果需要人工检查，只列出用户应该检查的页面和交互点，不要主动截图。

---

## 10. Git 与文件安全

禁止执行：

```bash
git reset --hard
git checkout .
git clean -fd
```

除非用户明确要求，否则不要：

- git commit
- git push
- revert 用户已有改动
- 批量删除文件或目录

删除文件规则：

1. 只能删除当前任务明确要求删除的单个文件。
2. 如果需要批量删除，停止操作并提示用户手动确认。
3. 不要删除用户未提交的文件。

---

## 11. 输出规则

完成任务后的回复保持简洁，格式如下：

```text
完成。

修改文件：
- xxx
- xxx

实现内容：
- xxx
- xxx

验证：
- xxx：通过 / 未运行，原因是 xxx

需要你手动处理：
- xxx
```

不要输出：

- 长篇背景解释
- 重复项目介绍
- 大量日志原文
- 完整 diff
- 无关建议
- 多轮计划
- 浏览器截图说明，除非用户要求

---

## 12. 节约 token 的特别规则

为了减少上下文和工具消耗，AI 必须遵守：

1. 不要重复读取已经明确知道的文件。
2. 不要一次性读取整个项目。
3. 不要在没有必要时运行测试。
4. 不要在没有必要时搜索全局代码。
5. 不要把命令输出完整复制到回复中。
6. 不要进行多方案长篇分析，除非用户要求。
7. 不要自动执行“计划 → 实现 → 测试 → 截图 → 再优化”的完整流程。
8. 如果任务很大，优先建议拆分，但不要阻塞当前最小可行修改。
9. 对 UI 任务，默认只做代码修改和 build，不做浏览器截图。
10. 对后端任务，默认只跑相关测试，进行逻辑验证就行，不跑全量测试。
